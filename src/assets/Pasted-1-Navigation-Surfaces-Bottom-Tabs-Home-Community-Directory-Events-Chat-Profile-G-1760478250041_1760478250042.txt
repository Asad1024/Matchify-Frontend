3) Live Event Mode (50-attendee flow)
	1.	Check-in: Scan event QR → unlock Live Mode → assign Table #
	2.	Rounds: 4–6 rounds × 15 min (10 swiping + 2 recap + 3 buffer)
	3.	Swiping: See curated queue (table first, then reshuffle). Left/Right → mutual like = Match → Chat/Meet
	4.	Reshuffle: AI re-groups tables; ensure new exposure and balance
	5.	No-Match Engagement: AI Wingmate, Mini-games, Second Chance window, Speed Coffee pairing, Leaderboard
	6.	Post-Event: Event Matches list, AI follow-ups, feedback, “Missed Connections” (upsell), invite to Community Lounge

4) Key User Journeys

4.1 New User → First Match

Sign up → Setup Wizard → Directory (Swipe) → Like/Match → Chat → Book coffee/event

4.2 RSVP → Live Event

See Events → RSVP/Pay → Ticket/QR → Check-in → Live Rounds → Match/Chat → Post-Event follow-ups

4.3 Community Engagement

Explore Groups → Join → Post/comment → Meet people → Convert to 1:1 match/event

5) Data Model (minimal)
	•	User{id, name, photos[], bio, tags[], intent, prefs, verified, membership, created_at}
	•	Compatibility: score_map[userId] = 0..100
	•	Post{id, author_id, group_id?, text, media[], likes, comments}
	•	Group{id, name, about, tags[], member_ids[]}
	•	Event{id, name, type, venue/online, time, capacity, price, ticket_rules, status}
	•	Ticket{id, event_id, user_id, qr_code, paid, checked_in}
	•	Round{id, event_id, start, end, table_assignments}
	•	Like{event_id?, from, to, ts}
	•	Match{id, a, b, event_id?, chat_id, created_at}
	•	Chat{id, participants[], messages[]}
	•	Payment{id, user_id, product, amount, status, provider_ref, created_at}
	•	Notification{id, user_id, type, payload, read}

6) Matching Logic (high level)
	•	Precompute compatibility from onboarding + quiz signals (values, interests, goals)
	•	Directory/Swipe: queue = highCompat(4) + medium(3) + diversity(2), dedupe seen_ids
	•	Live Mode: prioritize same-table profiles; after round, reshuffle to new table with unseen/high compat
	•	Second Chance: surface “people who liked you but you haven’t seen yet” (60s carousel)

7) Real-Time & Infra
	•	Realtime: WebSocket/Firebase channels:
	•	event:{eventId} (likes/matches/round timers)
	•	table:{tableId} (announcements)
	•	user:{userId} (personal nudges)
	•	Latency target: <200ms like→match popup
	•	Background jobs: round timers, reshuffle, leaderboard, post-event summaries
	•	Content safety: report/block, basic moderation

8) Notifications (examples)
	•	“Round 3 starts now—meet 8 new people.”
	•	“Aisha liked you—finish your queue to reveal.”
	•	“Speed Coffee at Table 3 in 2 minutes.”
	•	“You’ve got 2 new Event Matches—say hi!”

9) Permissions & Roles
	•	User: browse, like/match, chat, RSVP, post
	•	Premium: see match insights, voice/video, missed connections
	•	Host: start/pause round, push prompts, force reshuffle, view analytics
	•	Admin: event creation, refunds, moderation, content rules

10) Simple API Contract (illustrative)
	•	POST /auth/signup, POST /auth/verify
	•	GET /me, PATCH /me, PATCH /me/prefs
	•	GET /directory?filters, POST /like, GET /matches
	•	GET /events, POST /events/:id/rsvp, GET /tickets/:id/qrcode
	•	POST /events/:id/checkin, GET /events/:id/live, POST /events/:id/like
	•	GET /groups, POST /groups/:id/join, GET /groups/:id/feed
	•	GET /chat, POST /chat/:id/message
	•	GET /payments/tiers, POST /payments/subscribe
	•	GET /notifications, PATCH /notifications/read

11) Edge Cases & Rules
	•	Late arrivals → join next reshuffle
	•	Unbalanced tables → auto-rebalance priority
	•	Idle users → nudge with mini-games/coffee pairing
	•	Duplicate exposure → avoid showing same profile twice per event
	•	Safety → quick report & host review

12) Success Metrics
	•	Matches per attendee per round & total
	•	% attendees with ≥1 match
	•	Time-to-first-match
	•	Engagement score (swipes, chats, games)
	•	Post-event retention (chats continued, next event RSVPs)
	•	NPS/event rating
